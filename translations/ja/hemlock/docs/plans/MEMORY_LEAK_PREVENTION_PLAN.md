# メモリリーク防止計画

> Hemlockランタイムがメモリリークのない状態を確保し、プログラマとの契約を守ること。

**日付：** 2026-01-16
**ステータス：** 完了（v1.8.3で実装）
**バージョン：** 1.0

---

## エグゼクティブサマリー

Hemlockの設計思想は次の通りです：*「安全であるためのツールを提供しますが、使用を強制しません。」* これは**ランタイム自体**が、ユーザーコードがアンセーフな機能を使用する場合でもリークフリーでなければならないことを意味します。プログラマの契約は：

1. **ユーザー割り当て**（`alloc`、`buffer`）はプログラマの`free`の責任
2. **ランタイム内部割り当て**（文字列、配列、オブジェクト、クロージャ）は参照カウントで自動管理
3. **エラーと例外**はメモリをリークしてはならない
4. **非同期タスク**は明確な所有権セマンティクスを持つ
5. **ランタイムはプログラマから割り当てを隠さない**

この計画は現在のインフラの欠陥を特定し、体系的な改善を提案します。

## 特定されたギャップ

### ギャップ1：エラーパスのクリーンアップ（高優先度）

**問題：** 実行中に例外が発生した場合、割り当てられた一時オブジェクトがリークする可能性がある。

**シナリオ例：**
```hemlock
fn process_data() {
    let arr = [1, 2, 3];           // 配列が割り当てられる
    let transformed = arr.map(fn(x) {
        if (x == 2) { throw "error"; }  // 例外がスロー
        return x * 2;
    });
    // 'transformed'は部分的に割り当て、'arr'が解放されない可能性
}
```

### ギャップ2：デタッチされたタスク結果の所有権（中優先度）

**問題：** `detach(task)`はファイア・アンド・フォーゲット実行を許可するが、タスクの結果が回収されない可能性がある。

### ギャップ3：チャネルクローズ vs ドレインのセマンティクス（中優先度）

**問題：** バッファに値が残った状態でチャネルがクローズされた場合、それらの値は適切に解放されるか？

### ギャップ4：Null合体ASTリーク（修正済み）

**問題：** オプティマイザがnull合体式を最適化する際に、破棄されたASTノードを解放していなかった。

**修正：** 破棄されたノードを適切にクリーンアップするようにオプティマイザに追加。

### ギャップ5：クロージャキャプチャリストの粒度（低優先度）

**問題：** クロージャは参照された変数だけでなく環境チェーン全体をキャプチャし、ライフタイムを不必要に延長する可能性がある。

### ギャップ6：非同期コーディネーションでの循環参照（低優先度）

**問題：** タスクがチャネルを参照し、チャネルがタスクを参照するとサイクルが発生する可能性がある。

### ギャップ7：FFIメモリ境界のドキュメント（ドキュメント）

**問題：** FFI境界を越えた所有権移転が正式にドキュメント化されていない。

### ランタイム契約の検証

実装後に以下の保証が成立する必要がある：

1. **通常実行でリークなし**：有効なプログラムを実行して正常終了した場合、メモリリークなし（ランタイム内部）
2. **例外でリークなし**：例外のスローとキャッチでメモリリークなし
3. **タスク完了でリークなし**：完了したタスク（結合またはデタッチ）でメモリリークなし
4. **チャネルクローズでリークなし**：チャネルクローズ時にすべてのバッファ値を解放
5. **決定論的クリーンアップ**：デストラクタ呼び出しの順序が予測可能（deferはLIFO、オブジェクトはトポロジカル順）
