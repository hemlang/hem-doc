# コンパイラ最適化

Hemlockコンパイラ（`hemlockc`）はCコードを生成する際にいくつかの最適化パスを適用します。これらの最適化は自動的に行われ、ユーザーの介入は不要ですが、理解することでパフォーマンス特性の説明に役立ちます。

---

## 概要

```
ソース (.hml)
    ↓
  パース → AST
    ↓
  型チェック (オプション)
    ↓
  AST最適化パス
    ↓
  Cコード生成 (インライン化 + アンボックス化あり)
    ↓
  GCC/Clangコンパイル
```

---

## 式レベルのアンボックス化

Hemlockのランタイムはすべての値をタグ付き`HmlValue`構造体として表現します。インタプリタでは、すべての算術演算がランタイムディスパッチを通じて値のボックス化とアンボックス化を行います。コンパイラは、既知のプリミティブ型を持つ式に対してこのオーバーヘッドを排除します。

**最適化前（ナイーブなコード生成）：**
```c
// x + 1 (xはi32の場合)
hml_i32_add(hml_val_i32(x), hml_val_i32(1))  // 2回のボックス化 + ランタイムディスパッチ
```

**最適化後（式アンボックス化あり）：**
```c
// x + 1 (xはi32の場合)
hml_val_i32((x + 1))  // 純粋なC算術、最後に1回のボックス化
```

### アンボックス化されるもの

- 二項算術：`+`、`-`、`*`、`%`
- ビット演算：`&`、`|`、`^`、`<<`、`>>`
- 比較：`<`、`>`、`<=`、`>=`、`==`、`!=`
- 単項演算：`-`、`~`、`!`
- 型注釈付きの変数とループカウンタ

### HmlValueにフォールバックするもの

- 関数呼び出し（戻り値の型が動的な場合がある）
- 配列/オブジェクトアクセス（コンパイル時に要素型が不明）
- 型注釈がなく推論された型もない変数

### ヒント

ホットパスの変数に型注釈を追加すると、コンパイラがアンボックス化を適用しやすくなります：

```hemlock
// コンパイラはこの式全体をアンボックス化できる
fn dot(a: i32, b: i32, c: i32, d: i32): i32 {
    return a * c + b * d;
}
```

## マルチレベル関数インライン化

コンパイラは小さな関数を呼び出し元でインライン化し、関数呼び出しのオーバーヘッドを直接コードに置き換えます。Hemlockは深さ3までのマルチレベルインライン化をサポートしており、ネストされたヘルパー呼び出しもインライン化されます。

### 動作の仕組み

```hemlock
fn rotr(x: u32, n: i32): u32 => (x >> n) | (x << (32 - n));

fn ep0(x: u32): u32 => rotr(x, 2) ^ rotr(x, 13) ^ rotr(x, 22);

fn sha256_round(a: u32, ...): u32 {
    let s0 = ep0(a);  // ep0とrotrの両方がここでインライン化される
    // ...
}
```

深さ1では`ep0()`が`sha256_round()`にインライン化されます。深さ2では`ep0()`内の`rotr()`呼び出しもインライン化されます。結果は関数呼び出しのオーバーヘッドのない単一のネイティブ算術ブロックになります。

### インライン化の基準

関数は以下の場合にインライン化されます：
- 関数本体が小さい（単一の式または数個の文）
- 関数が再帰的でない
- 現在のインライン深度が3未満

### アノテーションによるインライン化の制御

```hemlock
@inline
fn always_inline(x: i32): i32 => x * 2;

@noinline
fn never_inline(x: i32): i32 {
    // 重複すべきでない複雑な関数
    return x;
}
```

## Whileループアキュムレータのアンボックス化

トップレベルのwhileループに対して、コンパイラはカウンタとアキュムレータ変数を検出し、ネイティブCローカル変数でシャドウすることで、各イテレーションでのボックス化/アンボックス化のオーバーヘッドを排除します。

### 最適化されるもの

```hemlock
let sum = 0;
let i = 0;
while (i < 1000000) {
    sum += i;
    i++;
}
print(sum);
```

コンパイラは`sum`と`i`がループ内でのみ使用される整数アキュムレータであることを検出し、`HmlValue`操作の代わりにネイティブの`int32_t`ローカル変数を生成します。これにより各イテレーションでのretain/releaseオーバーヘッドと型ディスパッチが排除されます。

### パフォーマンスへの影響

これらの最適化によるベンチマーク改善（一般的なワークロードで計測）：

| ベンチマーク | 最適化前 | 最適化後 | 改善率 |
|-----------|--------|-------|-------------|
| primes_sieve | 10ms | 6ms | -40% |
| binary_tree | 11ms | 8ms | -27% |
| json_serialize | 8ms | 5ms | -37% |
| json_deserialize | 10ms | 7ms | -30% |
| fibonacci | 29ms | 24ms | -17% |
| array_sum | 41ms | 36ms | -12% |

## ヘルパーアノテーション

コンパイラはGCC/Clang属性にマッピングされる10個の最適化アノテーションをサポートします：

| アノテーション | 効果 |
|------------|--------|
| `@inline` | 関数インライン化を推奨 |
| `@noinline` | 関数インライン化を防止 |
| `@hot` | 頻繁に実行されるとマーク（分岐予測） |
| `@cold` | まれに実行されるとマーク |
| `@pure` | 関数に副作用がない（外部状態を読み取る） |
| `@const` | 関数は引数のみに依存（外部状態なし） |
| `@flatten` | 関数内のすべての呼び出しをインライン化 |
| `@optimize(level)` | 関数ごとの最適化レベル（"0"-"3"、"s"、"fast"） |
| `@warn_unused` | 戻り値が無視された場合に警告 |
| `@section(name)` | 関数をカスタムELFセクションに配置 |

### 例

```hemlock
@hot @inline
fn fast_hash(key: string): u32 {
    // ホットパスのハッシュ関数
    let h: u32 = 5381;
    for (ch in key.chars()) {
        h = ((h << 5) + h) + ch;
    }
    return h;
}

@cold
fn handle_error(msg: string) {
    eprint("Error: " + msg);
    panic(msg);
}
```

## アロケーションプール

ランタイムは、頻繁に作成される短命のオブジェクトに対する`malloc`/`free`のオーバーヘッドを回避するために、事前割り当てされたオブジェクトプールを使用します：

| プール | スロット数 | 説明 |
|------|-------|-------------|
| 環境プール | 1024 | クロージャ/関数スコープ環境（各最大16変数） |
| オブジェクトプール | 512 | 最大8フィールドの匿名オブジェクト |
| 関数プール | 512 | キャプチャされた関数のクロージャ構造体 |

プールはO(1)の割り当てと解放のためにフリーリストスタックを使用します。プールが枯渇するとランタイムは`malloc`にフォールバックします。プールスロットを超えたオブジェクト（例：9番目のフィールドを追加したオブジェクト）は透過的にヒープストレージに移行されます。

### AST借用パラメータ

クロージャはディープコピーの代わりにASTから直接パラメータメタデータを借用し、クロージャ作成ごとに約6回の`malloc` + N回の`strdup`呼び出しを排除します。パラメータ名のハッシュはASTノード上で遅延計算されキャッシュされます。

## 型チェック

コンパイラにはコンパイル時型チェックが含まれています（デフォルトで有効）：

```bash
hemlockc program.hml -o program       # 型チェック + コンパイル
hemlockc --check program.hml          # 型チェックのみ
hemlockc --no-type-check program.hml  # 型チェックをスキップ
hemlockc --strict-types program.hml   # 暗黙の'any'型に警告
```

型注釈のないコードは動的（`any`型）として扱われ、常に型チェックをパスします。型注釈はアンボックス化を可能にする最適化ヒントを提供します。
