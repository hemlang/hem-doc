# План предотвращения утечек памяти

> Обеспечение отсутствия утечек памяти в среде выполнения Hemlock и соблюдение контракта с программистом.

**Дата:** 16.01.2026
**Статус:** Завершён (реализовано в v1.8.3)
**Версия:** 1.0

---

## Краткое содержание

Философия дизайна Hemlock гласит: *"Мы даём вам инструменты для безопасности, но не заставляем их использовать."* Это означает, что **сама среда выполнения** должна быть свободна от утечек, даже когда пользовательский код использует небезопасные возможности. Контракт программиста:

1. **Пользовательские выделения** (`alloc`, `buffer`) — ответственность программиста по вызову `free`
2. **Внутренние выделения среды выполнения** (строки, массивы, объекты, замыкания) — автоматическое управление через подсчёт ссылок
3. **Ошибки и исключения** не должны приводить к утечкам памяти
4. **Асинхронные задачи** имеют чёткую семантику владения
5. **Среда выполнения никогда не скрывает выделения** от программиста

Этот план определяет пробелы в текущей инфраструктуре и предлагает систематические улучшения.

## Выявленные пробелы

### Пробел 1: Очистка при ошибках (ВЫСОКИЙ ПРИОРИТЕТ)

**Проблема:** При возникновении исключений в середине выполнения выделенные временные объекты могут утекать.

**Пример:**
```hemlock
fn process_data() {
    let arr = [1, 2, 3];           // Массив выделен
    let transformed = arr.map(fn(x) {
        if (x == 2) { throw "error"; }  // Исключение
        return x * 2;
    });
    // 'transformed' частично выделен, 'arr' может не быть освобождён
}
```

### Пробел 2: Владение результатами отсоединённых задач (СРЕДНИЙ ПРИОРИТЕТ)

**Проблема:** `detach(task)` позволяет выполнение "запустил и забыл", но результат задачи может никогда не быть собран.

### Пробел 3: Семантика close vs. drain каналов (СРЕДНИЙ ПРИОРИТЕТ)

**Проблема:** При закрытии канала с оставшимися буферизованными значениями — правильно ли они освобождаются?

### Пробел 4: Утечка AST при null-объединении (ИСПРАВЛЕНО)

**Проблема:** Оптимизатор оптимизировал выражения null-объединения, когда результат был известен во время компиляции, но не освобождал отброшенные узлы AST.

### Пробел 5: Гранулярность захвата замыканий (НИЗКИЙ ПРИОРИТЕТ)

**Проблема:** Замыкания захватывают всю цепочку окружений, а не только ссылаемые переменные, потенциально продлевая время жизни без необходимости.

### Пробел 6: Циклические ссылки в асинхронной координации (НИЗКИЙ ПРИОРИТЕТ)

**Проблема:** Задачи, ссылающиеся на каналы, которые ссылаются на задачи, могут создавать циклы.

### Пробел 7: Документация границ FFI-памяти (ДОКУМЕНТАЦИЯ)

**Проблема:** Передача владения через границу FFI формально не документирована.

## Предложенные улучшения

### Фаза 1: Критические исправления (Недели 1-2)

- Безопасная относительно исключений оценка выражений (стек временных значений)
- Очистка результатов отсоединённых задач
- Дренаж канала при закрытии

### Фаза 2: Исправления известных проблем (Недели 3-4)

- Исправление AST null-объединения
- Оптимизация захвата замыканий (опционально)

### Фаза 3: Укрепление инфраструктуры тестирования (Недели 5-6)

- Набор регрессионных тестов утечек
- Непрерывный мониторинг утечек
- Фаззинг-тестирование на безопасность памяти

### Фаза 4: Документация и контракт (Неделя 7)

- Документация владения памятью
- Правила владения FFI

## Критерии успеха

### Количественные

- [ ] Ноль утечек по отчётам ASAN на полном наборе тестов
- [ ] Ноль утечек по отчётам Valgrind на полном наборе тестов
- [ ] Базовая линия утечек установлена и контролируется в CI
- [ ] 100% выявленных пробелов устранены или документированы как допустимые

### Гарантии контракта среды выполнения

1. **Нет утечек при нормальном выполнении**
2. **Нет утечек при исключениях**
3. **Нет утечек при завершении задач** (присоединённых или отсоединённых)
4. **Нет утечек при закрытии каналов**
5. **Детерминированная очистка**
