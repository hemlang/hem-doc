# HemlockScript: Hemlock → WASM через Emscripten

> Переносимые программы Hemlock, работающие в браузере.

## Цель

Добавить цель компиляции WASM в `hemlockc`, чтобы программы Hemlock могли работать в браузерах и других WASM-средах (Node/Deno/Cloudflare Workers). Подход: Hemlock → C (существующий конвейер) → WASM (через Emscripten), с прослойкой совместимости для браузера, заменяющей POSIX-зависимые встроенные функции.

**Вне области:** Переписывание компилятора или среды выполнения с нуля. Мы используем существующую генерацию C-кода `hemlockc` и `libhemlock_runtime` максимально.

---

## Архитектура

```
Исходный код Hemlock (.hml)
        ↓
   hemlockc (существующий frontend + codegen)
        ↓
   Сгенерированный C-код (существующий)
        ↓
   emcc (Emscripten)  ←  libhemlock_runtime_wasm.a (адаптированная среда выполнения)
        ↓
   program.wasm + program.js (загрузчик/связка)
        ↓
   Браузер / Node / Deno / WASM-среда
```

Ключевое понимание: `hemlockc` уже генерирует переносимый C. Нам не нужен новый бэкенд — нам нужна WASM-совместимая библиотека среды выполнения и конвейер сборки Emscripten.

---

## Фаза 1: Минимальная WASM-сборка (только ядро языка)

**Результат:** `make wasm` создаёт связку `.wasm` + `.js`, которая может запускать чисто вычислительные программы Hemlock в браузере.

- Создание WASM-прослойки среды выполнения (`runtime/src/wasm_shim.c`)
- Добавление `#ifdef __EMSCRIPTEN__` защит в среду выполнения
- Создание WASM-специфичной цели Makefile
- Модификация `hemlockc` для поддержки цели WASM (`--target wasm`)
- Создание HTML-тестовой страницы

## Фаза 2: Браузерный ввод/вывод и stdlib

**Результат:** Программы Hemlock могут выполнять полезную работу в браузере — доступ к файлам (виртуальная ФС), операции со временем, и переносимые модули stdlib работают.

- Виртуальная файловая система Emscripten (MEMFS/IDBFS)
- Портирование 22 уже переносимых модулей stdlib
- Адаптация модулей времени и окружения

## Фаза 3: Мост JavaScript-интеропа

**Результат:** WASM-программы Hemlock могут вызывать API браузера и функции JavaScript, а JavaScript может вызывать функции Hemlock.

- Мост JS ↔ Hemlock для двунаправленных вызовов функций
- Модуль `@wasm/browser` для API браузера
- HTTP через fetch, WebSocket через браузерный WebSocket
- Крипто через Web Crypto API

## Фаза 4: Async и потоки (расширение)

**Результат:** `spawn`/`await` Hemlock работает в браузере с использованием Web Workers.

- Порождение задач на основе Web Workers
- Asyncify для блокирующих операций

---

## Что работает сразу (без изменений)

Эти возможности Hemlock компилируются в стандартный C, который Emscripten обрабатывает нативно:

- Вся арифметика, побитовые, логические операторы
- Переменные, области видимости, замыкания
- Функции, рекурсия, функции с телом-выражением
- if/else, while, for, loop, switch, сопоставление с образцом
- Объекты, массивы, строки (все 19+23 метода)
- Аннотации типов и проверка типов времени выполнения
- Try/catch/finally/throw
- Defer
- Шаблонные строки
- Null-объединение (`??`, `?.`, `??=`)
- Именованные аргументы
- Составные типы, псевдонимы типов
- `print()`, `eprint()` (через маппинг консоли Emscripten)
- `alloc()`/`free()`/`buffer()` (линейная память)
- `typeof()`, `len()`, `sizeof()`
- Математические встроенные (sin, cos, sqrt и др.)
- Весь подсчёт ссылок / управление памятью

## Оценка объёма по фазам

| Фаза | Объём | Зависимости |
|------|-------|-------------|
| **Фаза 1** | ~800 строк нового C, ~200 строк #ifdef защит, изменения Makefile | Emscripten SDK |
| **Фаза 2** | ~200 строк C, тестирование stdlib, Makefile | Фаза 1 |
| **Фаза 3** | ~600 строк C (мост), ~200 строк Hemlock (stdlib) | Фаза 1 |
| **Фаза 4** | ~1000 строк C (потоки Worker), сложно | Фаза 1+3 |

Рекомендуемый порядок: Фаза 1 → Фаза 2 → Фаза 3 → Фаза 4

Одна только Фаза 1 даёт полезный "Hemlock в браузере" для вычислительных задач. Фазы 2-4 постепенно добавляют ввод/вывод и интероп.
