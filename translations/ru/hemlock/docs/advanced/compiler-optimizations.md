# Оптимизации компилятора

Компилятор Hemlock (`hemlockc`) применяет несколько проходов оптимизации при генерации C-кода. Эти оптимизации выполняются автоматически и не требуют вмешательства пользователя, но понимание их работы помогает объяснить характеристики производительности.

---

## Обзор

```
Исходный код (.hml)
    ↓
  Парсинг → AST
    ↓
  Проверка типов (опционально)
    ↓
  Проход оптимизации AST
    ↓
  Генерация C-кода (с инлайнингом + распаковкой)
    ↓
  Компиляция GCC/Clang
```

---

## Распаковка на уровне выражений

Среда выполнения Hemlock представляет все значения как теговые структуры `HmlValue`. В интерпретаторе каждая арифметическая операция упаковывает и распаковывает значения через диспетчеризацию времени выполнения. Компилятор устраняет эти накладные расходы для выражений с известными примитивными типами.

**До (наивная генерация кода):**
```c
// x + 1 где x имеет тип i32
hml_i32_add(hml_val_i32(x), hml_val_i32(1))  // 2 вызова упаковки + диспетчеризация
```

**После (с распаковкой выражений):**
```c
// x + 1 где x имеет тип i32
hml_val_i32((x + 1))  // Чистая арифметика C, одна упаковка в конце
```

### Что распаковывается

- Бинарная арифметика: `+`, `-`, `*`, `%`
- Побитовые операции: `&`, `|`, `^`, `<<`, `>>`
- Сравнения: `<`, `>`, `<=`, `>=`, `==`, `!=`
- Унарные операции: `-`, `~`, `!`
- Переменные с аннотациями типов и счётчики циклов

### Что остаётся в HmlValue

- Вызовы функций (возвращаемый тип может быть динамическим)
- Доступ к массивам/объектам (тип элемента неизвестен во время компиляции)
- Переменные без аннотаций типов и без выведенного типа

### Совет

Добавление аннотаций типов к переменным на горячем пути помогает компилятору применять распаковку:

```hemlock
// Компилятор может распаковать всё выражение
fn dot(a: i32, b: i32, c: i32, d: i32): i32 {
    return a * c + b * d;
}
```

## Многоуровневый инлайнинг функций

Компилятор встраивает небольшие функции в места вызова, заменяя накладные расходы вызова функции прямым кодом. Hemlock поддерживает многоуровневый инлайнинг до глубины 3, что означает, что вложенные вспомогательные вызовы также встраиваются.

### Как это работает

```hemlock
fn rotr(x: u32, n: i32): u32 => (x >> n) | (x << (32 - n));

fn ep0(x: u32): u32 => rotr(x, 2) ^ rotr(x, 13) ^ rotr(x, 22);

fn sha256_round(a: u32, ...): u32 {
    let s0 = ep0(a);  // И ep0, И rotr встраиваются здесь
    // ...
}
```

На глубине 1 `ep0()` встраивается в `sha256_round()`. На глубине 2 вызовы `rotr()` внутри `ep0()` также встраиваются. Результат — единый блок нативной арифметики без накладных расходов на вызовы функций.

### Критерии инлайнинга

Функции встраиваются когда:
- Тело функции маленькое (одно выражение или несколько операторов)
- Функция не рекурсивная
- Текущая глубина инлайнинга меньше 3

### Управление инлайнингом с помощью аннотаций

```hemlock
@inline
fn always_inline(x: i32): i32 => x * 2;

@noinline
fn never_inline(x: i32): i32 {
    // Сложная функция, которая не должна дублироваться
    return x;
}
```

## Распаковка аккумуляторов циклов while

Для циклов while верхнего уровня компилятор обнаруживает переменные-счётчики и аккумуляторы и создаёт для них теневые нативные локальные переменные C, устраняя накладные расходы упаковки/распаковки на каждой итерации.

### Что оптимизируется

```hemlock
let sum = 0;
let i = 0;
while (i < 1000000) {
    sum += i;
    i++;
}
print(sum);
```

Компилятор определяет, что `sum` и `i` — целочисленные аккумуляторы, используемые только внутри цикла, и генерирует нативные `int32_t` локальные переменные вместо операций с `HmlValue`. Это устраняет накладные расходы retain/release и диспетчеризацию типов на каждой итерации.

### Влияние на производительность

Улучшения бенчмарков от этих оптимизаций (измерены на типичных нагрузках):

| Бенчмарк | До | После | Улучшение |
|-----------|------|-------|-----------|
| primes_sieve | 10мс | 6мс | -40% |
| binary_tree | 11мс | 8мс | -27% |
| json_serialize | 8мс | 5мс | -37% |
| json_deserialize | 10мс | 7мс | -30% |
| fibonacci | 29мс | 24мс | -17% |
| array_sum | 41мс | 36мс | -12% |

## Вспомогательные аннотации

Компилятор поддерживает 10 аннотаций оптимизации, которые отображаются на атрибуты GCC/Clang:

| Аннотация | Эффект |
|-----------|--------|
| `@inline` | Поощрять встраивание функции |
| `@noinline` | Предотвратить встраивание функции |
| `@hot` | Отметить как часто выполняемую (предсказание ветвлений) |
| `@cold` | Отметить как редко выполняемую |
| `@pure` | Функция без побочных эффектов (читает внешнее состояние) |
| `@const` | Функция зависит только от аргументов (нет внешнего состояния) |
| `@flatten` | Встроить все вызовы внутри функции |
| `@optimize(level)` | Уровень оптимизации для функции ("0"-"3", "s", "fast") |
| `@warn_unused` | Предупреждать, если возвращаемое значение проигнорировано |
| `@section(name)` | Разместить функцию в пользовательской секции ELF |

### Пример

```hemlock
@hot @inline
fn fast_hash(key: string): u32 {
    // Функция хеширования на горячем пути
    let h: u32 = 5381;
    for (ch in key.chars()) {
        h = ((h << 5) + h) + ch;
    }
    return h;
}

@cold
fn handle_error(msg: string) {
    eprint("Error: " + msg);
    panic(msg);
}
```

## Пулы выделения

Среда выполнения использует предварительно выделенные пулы объектов для избежания накладных расходов `malloc`/`free` для часто создаваемых короткоживущих объектов:

| Пул | Слоты | Описание |
|-----|-------|----------|
| Пул окружений | 1024 | Окружения замыканий/функций (до 16 переменных каждое) |
| Пул объектов | 512 | Анонимные объекты с не более 8 полями |
| Пул функций | 512 | Структуры замыканий для захваченных функций |

Пулы используют стеки свободных списков для O(1) выделения и освобождения. Когда пул исчерпан, среда выполнения переходит к `malloc`. Объекты, которые перерастают свой слот в пуле (например, объект получает 9-е поле), прозрачно мигрируют в кучу.

### Заимствование параметров из AST

Замыкания заимствуют метаданные параметров непосредственно из AST вместо глубокого копирования, устраняя примерно 6 вызовов `malloc` + N вызовов `strdup` при создании каждого замыкания. Хеши имён параметров лениво вычисляются и кэшируются в узле AST.

## Проверка типов

Компилятор включает проверку типов во время компиляции (включена по умолчанию):

```bash
hemlockc program.hml -o program       # Проверка типов + компиляция
hemlockc --check program.hml          # Только проверка типов
hemlockc --no-type-check program.hml  # Пропустить проверку типов
hemlockc --strict-types program.hml   # Предупреждать о неявных типах 'any'
```

Нетипизированный код рассматривается как динамический (тип `any`) и всегда проходит проверку типов. Аннотации типов предоставляют подсказки оптимизации, позволяющие распаковку.
