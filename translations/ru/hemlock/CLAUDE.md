# Философия дизайна языка Hemlock

> "Маленький небезопасный язык для безопасного написания небезопасных вещей."

Этот документ содержит основные принципы проектирования для ИИ-ассистентов, работающих с Hemlock.
Подробную документацию см. в `docs/README.md` и каталоге `stdlib/docs/`.

---

## Основная идентичность

Hemlock — это **системный скриптовый язык** с ручным управлением памятью и явным контролем:
- Мощь C с современной эргономикой скриптовых языков
- Встроенная структурированная асинхронная конкурентность
- Никакого скрытого поведения или магии

**Hemlock НЕ является:** Безопасным по памяти, языком с GC или скрывающим сложность.
**Hemlock ЯВЛЯЕТСЯ:** Явным, а не неявным, образовательным, "слоем C-скриптов" для системной работы.

## Принципы дизайна

### 1. Явное вместо неявного
- Точки с запятой обязательны (нет ASI)
- Ручное управление памятью (alloc/free)
- Аннотации типов опциональны, но проверяются во время выполнения

### 2. Динамический по умолчанию, типизированный по выбору
- Каждое значение имеет тег типа времени выполнения
- Литералы выводят типы: `42` → i32, `5000000000` → i64, `3.14` → f64
- Опциональные аннотации типов обеспечивают проверки времени выполнения

### 3. Небезопасность — это возможность
- Арифметика указателей разрешена (ответственность пользователя)
- Нет проверки границ для сырого `ptr` (используйте `buffer` для безопасности)
- Допускаются сбои из-за двойного освобождения

### 4. Структурированная конкурентность первого класса
- Встроенные `async`/`await` с параллелизмом на основе pthread
- Каналы для коммуникации
- `spawn`/`join`/`detach` для управления задачами

### 5. C-подобный синтаксис
- Блоки `{}` всегда обязательны
- Комментарии: `// строка` и `/* блок */`
- Операторы как в C: `+`, `-`, `*`, `%`, `&&`, `||`, `!`, `&`, `|`, `^`, `<<`, `>>`
- Инкремент/декремент: `++x`, `x++`, `--x`, `x--` (префиксный и постфиксный)
- Составное присваивание: `+=`, `-=`, `*=`, `/=`, `%=`, `&=`, `|=`, `^=`, `<<=`, `>>=`
- `/` всегда возвращает float (используйте `divi()` для целочисленного деления)
- Синтаксис типов: `let x: type = value;`

### Типы
```
Знаковые:      i8, i16, i32, i64
Беззнаковые:   u8, u16, u32, u64
С плавающей точкой: f32, f64
Другие:        bool, string, rune, array, ptr, buffer, null, object, file, task, channel
```

### Операторы объединения с null
```hemlock
// Объединение с null (??) - возвращает левое если не null, иначе правое
let name = user.name ?? "Аноним";
let first = a ?? b ?? c ?? "запасной";

// Присваивание с объединением с null (??=) - присваивает только если null

config ??= { timeout: 60 };    // config не изменился (не null)

// Работает со свойствами и индексами

arr[0] ??= "первый";

// Безопасная навигация (?.) - возвращает null если объект null
let city = user?.address?.city;  // null если любая часть null
let upper = name?.to_upper();    // безопасный вызов метода
let item = arr?.[0];             // безопасная индексация
```

### Именованные аргументы
```hemlock
// Функции могут вызываться с именованными аргументами
fn create_user(name: string, age?: 18, active?: true) {
    print(name + " имеет возраст " + age + " лет");
}

// Позиционные аргументы (традиционные)
create_user("Алиса", 25, false);

// Именованные аргументы - могут быть в любом порядке
create_user(name: "Борис", age: 30);
create_user(age: 25, name: "Чарли", active: false);

// Пропуск опциональных параметров через именование нужных
create_user("Дмитрий", active: false);  // Использует age=18 по умолчанию

// Именованные аргументы должны идти после позиционных
create_user("Ева", age: 21);          // OK: сначала позиционный, потом именованный
// create_user(name: "Плохо", 25);    // ОШИБКА: позиционный после именованного
```

**Правила:**
- Именованные аргументы используют синтаксис `name: value`
- Могут появляться в любом порядке после позиционных аргументов
- Позиционные аргументы не могут следовать за именованными
- Работают с параметрами по умолчанию/опциональными
- Неизвестные имена параметров вызывают ошибки времени выполнения

### Составные типы (пересечение/утиная типизация)
```hemlock
// Определение структурных типов
define HasName { name: string }
define HasAge { age: i32 }
define HasEmail { email: string }

// Составной тип: объект должен удовлетворять ВСЕМ типам
let person: HasName & HasAge = { name: "Алиса", age: 30 };

// Параметры функции с составными типами
fn greet(p: HasName & HasAge) {
    print(p.name + " имеет возраст " + p.age);
}

// Три или более типов
fn describe(p: HasName & HasAge & HasEmail) {
    print(p.name + " <" + p.email + ">");
}

// Дополнительные поля разрешены (утиная типизация)
let employee: HasName & HasAge = {
    name: "Борис",
    age: 25,
    department: "Инженерия"  // OK - дополнительные поля игнорируются
};
```

Составные типы обеспечивают поведение, подобное интерфейсам, без отдельного ключевого слова `interface`,
основываясь на существующих парадигмах `define` и утиной типизации.

### Const-параметры
```hemlock
// Const-параметр - глубокая неизменяемость
fn print_all(const items: array) {
    // items.push(4);  // ОШИБКА: нельзя изменять const-параметр

// Const с объектами - никаких изменений через любой путь

    // person.name = "Борис"; // ОШИБКА: нельзя изменять
```

Модификатор `const` предотвращает любое изменение параметра, включая вложенные свойства.
Это обеспечивает безопасность во время компиляции для функций, которые не должны модифицировать свои входные данные.

### Ref-параметры (передача по ссылке)
```hemlock
// Ref-параметр - переменная вызывающего кода модифицируется напрямую
fn increment(ref x: i32) {
    x = x + 1;  // Модифицирует оригинальную переменную
}

let count = 10;
increment(count);
print(count);  // 11 - оригинал был изменён

// Классическая функция swap
fn swap(ref a: i32, ref b: i32) {
    let temp = a;
    a = b;
    b = temp;
}

let x = 1;
let y = 2;
swap(x, y);
print(x, y);  // 2 1

// Смешивание ref и обычных параметров
fn add_to(ref target: i32, amount: i32) {
    target = target + amount;
}

let total = 100;
add_to(total, 50);
print(total);  // 150
```

Модификатор `ref` передаёт ссылку на переменную вызывающего кода, позволяя функции
модифицировать её напрямую. Без `ref` примитивы передаются по значению (копируются). Используйте `ref`, когда
нужно изменить состояние вызывающего кода без возврата значения.

**Правила:**
- В `ref`-параметры должны передаваться переменные, не литералы или выражения
- Работает со всеми типами (примитивы, массивы, объекты)
- Комбинируется с аннотациями типов: `ref x: i32`
- Нельзя комбинировать с `const` (они противоположны)

### Сигнатуры методов в define
```hemlock
// Define с сигнатурами методов (паттерн интерфейса)
define Comparable {
    value: i32,
    fn compare(other: Self): i32   // Требуемая сигнатура метода
}

// Объекты должны предоставлять требуемый метод
let a: Comparable = {
    value: 10,
    compare: fn(other) { return self.value - other.value; }
};

// Опциональные методы с ?
define Serializable {
    fn serialize(): string,        // Обязательный
    fn pretty?(): string           // Опциональный метод
}

// Тип Self ссылается на определяющий тип
define Cloneable {
    fn clone(): Self   // Возвращает тот же тип, что и объект
}
```

Сигнатуры методов в блоках `define` используют разделители-запятые (как интерфейсы TypeScript),
устанавливая контракты, которые объекты должны выполнять, и позволяя программирование в стиле интерфейсов
с системой утиной типизации Hemlock.

### Async/Конкурентность
```hemlock
async fn compute(n: i32): i32 { return n * n; }
let task = spawn(compute, 42);
let result = await task;     // или join(task)
detach(spawn(background_work));

let ch = channel(10);
ch.send(value);
let val = ch.recv();
ch.close();
```

**Владение памятью:** Задачи получают копии примитивных значений, но разделяют указатели. Если вы передаёте `ptr` в порождённую задачу, вы должны убедиться, что память остаётся действительной до завершения задачи. Используйте `join()` перед `free()`, или используйте каналы для сигнализации о завершении.

```hemlock
// every(predicate) - true если все элементы удовлетворяют предикату
let allPositive = [1, 2, 3].every(fn(x) { return x > 0; });  // true

// some(predicate) - true если хотя бы один элемент удовлетворяет предикату
let hasEven = [1, 2, 3].some(fn(x) { return x % 2 == 0; });  // true

// indexOf(value) - находит первый индекс значения, или -1
let idx = ["a", "b", "c"].indexOf("b");  // 1

// sort(comparator?) - сортировка на месте, необязательный компаратор
let nums = [3, 1, 4, 1, 5];
nums.sort();                              // [1, 1, 3, 4, 5]
nums.sort(fn(a, b) { return b - a; });    // по убыванию

// fill(value, start?, end?) - заполняет значением
let arr = [1, 2, 3, 4, 5];
arr.fill(0);        // [0, 0, 0, 0, 0]
arr.fill(9, 2);     // [0, 0, 9, 9, 9]
arr.fill(7, 1, 4);  // [0, 7, 7, 7, 9]
```

Типизированные массивы: `let nums: array<i32> = [1, 2, 3];`

## FFI (Интерфейс внешних функций)

Объявление и вызов C-функций из общих библиотек:
```hemlock
import "libc.so.6";

extern fn strlen(s: string): i32;
extern fn getpid(): i32;

let len = strlen("Привет!");  // 13
let pid = getpid();
```

Экспорт FFI-функций из модулей:
```hemlock
// string_utils.hml
import "libc.so.6";

export extern fn strlen(s: string): i32;
export fn string_length(s: string): i32 {
    return strlen(s);
}
```

Динамический FFI (связывание во время выполнения):
```hemlock
let lib = ffi_open("libc.so.6");
let puts = ffi_bind(lib, "puts", [FFI_POINTER], FFI_INT);
puts("Привет из C!");
ffi_close(lib);
```

Типы: `FFI_INT`, `FFI_DOUBLE`, `FFI_POINTER`, `FFI_STRING`, `FFI_VOID` и др.

## Атомарные операции

Lock-free конкурентное программирование с атомарными операциями:

```hemlock
// Выделение памяти для атомарного i32
let p = alloc(4);
ptr_write_i32(p, 0);

// Атомарная загрузка/сохранение
let val = atomic_load_i32(p);        // Атомарное чтение
atomic_store_i32(p, 42);             // Атомарная запись

// Операции fetch-and-modify (возвращают СТАРОЕ значение)
let old = atomic_add_i32(p, 10);     // Сложение, возврат старого
old = atomic_sub_i32(p, 5);          // Вычитание, возврат старого
old = atomic_and_i32(p, 0xFF);       // Побитовое И
old = atomic_or_i32(p, 0x10);        // Побитовое ИЛИ
old = atomic_xor_i32(p, 0x0F);       // Побитовое XOR

// Compare-and-swap (CAS)
let success = atomic_cas_i32(p, 42, 100);  // Если *p == 42, установить 100
// Возвращает true если swap успешен, иначе false

// Атомарный обмен
old = atomic_exchange_i32(p, 999);   // Обмен, возврат старого

free(p);

// Доступны варианты i64 (atomic_load_i64, atomic_add_i64 и др.)

// Барьер памяти (полный барьер)
atomic_fence();
```

Все операции используют последовательную согласованность (`memory_order_seq_cst`).

## Рекомендации по стилю кода

### Константы и магические числа

При добавлении числовых констант в кодовую базу C следуйте этим рекомендациям:

1. **Определяйте константы в `include/hemlock_limits.h`** - Этот файл является центральным местом для всех ограничений времени компиляции и выполнения, ёмкостей и именованных констант.

2. **Используйте описательные имена с префиксом `HML_`** - Все константы должны иметь префикс `HML_` для ясности пространства имён.

3. **Избегайте магических чисел** - Заменяйте жёстко закодированные числовые значения именованными константами. Примеры:
   - Пределы диапазонов типов: `HML_I8_MIN`, `HML_I8_MAX`, `HML_U32_MAX`
   - Ёмкости буферов: `HML_INITIAL_ARRAY_CAPACITY`, `HML_INITIAL_LEXER_BUFFER_CAPACITY`
   - Преобразования времени: `HML_NANOSECONDS_PER_SECOND`, `HML_MILLISECONDS_PER_SECOND`
   - Seed-значения хешей: `HML_DJB2_HASH_SEED`
   - ASCII-значения: `HML_ASCII_CASE_OFFSET`, `HML_ASCII_PRINTABLE_START`

4. **Включайте `hemlock_limits.h`** - Исходные файлы должны включать этот заголовок (часто через `internal.h`) для доступа к константам.

5. **Документируйте назначение** - Добавляйте комментарий, объясняющий, что представляет каждая константа.

```
└────────────┘    └────────────┘
```

### Проверка типов компилятора

Компилятор (`hemlockc`) включает проверку типов во время компиляции, **включённую по умолчанию**:

```bash
hemlockc program.hml -o program    # Проверка типов, затем компиляция
hemlockc --check program.hml       # Только проверка типов, без компиляции
hemlockc --no-type-check prog.hml  # Отключить проверку типов
hemlockc --strict-types prog.hml   # Предупреждать о неявных типах 'any'
```

Проверка типов:
- Валидирует аннотации типов во время компиляции
- Рассматривает нетипизированный код как динамический (тип `any`) - всегда валиден
- Предоставляет подсказки оптимизации для распаковки
- Использует разрешительные числовые преобразования (диапазон проверяется во время выполнения)

## Разработка с приоритетом паритета

**И интерпретатор, и компилятор должны производить идентичный вывод для одного и того же входа.**

### Политика разработки

При добавлении или изменении возможностей языка:

1. **Проектирование** - Определите изменение AST/семантики в общем frontend'е
2. **Реализация интерпретатора** - Добавьте вычисление обходом дерева
3. **Реализация компилятора** - Добавьте генерацию C-кода
4. **Добавление теста паритета** - Напишите тест в `tests/parity/` с файлом `.expected`
5. **Проверка** - Запустите `make parity` перед слиянием

### Структура тестов паритета

```
tests/parity/
├── language/       # Основные возможности языка (управление потоком, замыкания и др.)
├── builtins/       # Встроенные функции (print, typeof, memory и др.)
├── methods/        # Методы строк и массивов
└── modules/        # Import/export, импорты stdlib
```

Каждый тест имеет два файла:
- `feature.hml` - Тестовая программа
- `feature.expected` - Ожидаемый вывод (должен совпадать для обоих backend'ов)

### Результаты тестов паритета

| Статус | Значение |
|--------|----------|
| `✓ PASSED` | И интерпретатор, и компилятор соответствуют ожидаемому выводу |
| `◐ INTERP_ONLY` | Интерпретатор работает, компилятор не работает (нужно исправить компилятор) |
| `◑ COMPILER_ONLY` | Компилятор работает, интерпретатор не работает (редко) |
| `✗ FAILED` | Оба не работают (ошибка теста или реализации) |

### Что требует паритета

- Все конструкции языка (if, while, for, switch, defer, try/catch)
- Все операторы (арифметические, побитовые, логические, сравнения)
- Все встроенные функции (print, typeof, alloc и др.)
- Все методы строк и массивов
- Правила приведения и продвижения типов
- Сообщения об ошибках для ошибок времени выполнения

### Что может отличаться

- Характеристики производительности
- Детали расположения в памяти
- Формат отладки/стека вызовов
- Ошибки компиляции (компилятор может обнаружить больше во время компиляции)

### Добавление теста паритета

```bash
# 1. Создание тестового файла
cat > tests/parity/language/my_feature.hml << 'EOF'
// Описание теста
let x = some_feature();
print(x);
EOF

# 2. Генерация ожидаемого вывода из интерпретатора
./hemlock tests/parity/language/my_feature.hml > tests/parity/language/my_feature.expected

# 3. Проверка паритета
make parity
```

## Версия

**v1.9.2** - Текущий релиз с:
- **Исправление unboxed-счётчика цикла компилятора** - Исправлен критический баг codegen, при котором оптимизированные счётчики циклов (нативные `int32_t`) использовались напрямую как инициализаторы `HmlValue` без boxing. Проверка `codegen_is_main_var` ошибочно предотвращала генерацию boxing-обёртки (`hml_val_i32()`), когда переменная основного уровня перекрывала unboxed-счётчик цикла внутри функции модуля/замыкания. Исправляет компиляцию `@stdlib/collections` и `@stdlib/encoding`.
- **Диспатч объектного метода `clear()`** - Компилятор теперь корректно диспатчит `.clear()` на объектные методы при вызове на не-массивных типах. Ранее `.clear()` всегда генерировал `hml_array_clear()` независимо от типа получателя.
- **Исправление затенения импорта `exec()`** - Встроенный обработчик `exec()` компилятора теперь проверяет привязки импорта и локальные функции модуля перед диспатчем к системному встроенному exec. Исправляет `@stdlib/sqlite`, который экспортирует собственную функцию `exec()`.

**v1.9.1** - Предыдущий релиз с:
- **Встроенная функция `write()`** - Вывод без завершающего перевода строки (`write("hello"); write(" world");` выводит в одну строку). Включает `fflush(stdout)` для немедленного вывода. Полный паритет между интерпретатором и компилятором.
- **Одноаргументный `slice()`** - `arr.slice(n)` и `str.slice(n)` теперь по умолчанию устанавливают конец на длину, соответствуя поведению JS/Python. Двухаргументная форма не изменена.
- **`join()` на массивах рун** - `"hello".chars().join("")` теперь корректно выдаёт `"hello"` вместо `"[object][object]..."`. Позволяет идиоматическое обращение строки: `str.chars().reverse().join("")`.
- **Приведение числовых ключей HashMap** - Ключи разных числовых типов теперь совпадают (например, ключ `i32` найден при поиске `i64`). Ранее guard `typeof()` в `keys_equal()` отклонял валидные кросс-типные совпадения.
- **Улучшения HemBench** - Исправлены определения задач (округление L1-M-02, точность L2-E-01), прекращена утечка ожидаемого вывода в задачи бенчмарка L5/L6.
- **Полные встроенные функции `ptr_read_*`** - Добавлены `ptr_read_i8`, `ptr_read_i16`, `ptr_read_i64`, `ptr_read_u8`, `ptr_read_u16`, `ptr_read_u32`, `ptr_read_u64`, `ptr_read_f32`, `ptr_read_f64`, `ptr_read_ptr` в дополнение к существующим функциям `ptr_write_*`. Исправлен `ptr_read_i32` на прямое разыменование (было двойное разыменование). Полный паритет между интерпретатором и компилятором.
- **Загрузка FFI-библиотек на macOS** - `dlopen` теперь ищет в `/usr/local/lib` и `/opt/homebrew/lib` как запасные пути на macOS, исправляя ошибки «библиотека не найдена» для установленных пользователем разделяемых библиотек (например, libusearch_c).

  - Безопасная относительно исключений оценка выражений (массивы, объекты, вызовы функций)
  - Корректный retain/release результата задачи при join()

- **Вспомогательные аннотации компилятора** - 11 аннотаций оптимизации для управления GCC/Clang:

## Философия

> Мы даём вам инструменты для безопасности (`buffer`, аннотации типов, проверка границ), но не заставляем их использовать (`ptr`, ручная память, небезопасные операции).

**Если вы не уверены, подходит ли возможность для Hemlock, спросите: "Это даёт программисту больше явного контроля или это что-то скрывает?"**

Если скрывает, то, вероятно, это не место в Hemlock.
